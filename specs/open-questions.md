## 2026-05-29 · Vision 初稿完成，Spec 調整前的前置問題

完成 `vision.md` 初稿後，由 AI agent 檢視文件所識別的 5 個缺口。在調整現有 spec 或新增 spec 前需要先解決。

---

### ✅ 1. QRCode 機制：房間 vs. 任務（優先級：最高）

**問題**：現有 spec 的架構是「掃碼加入房間，房間內有多個登記項目」。vision.md 的成功場景描述的是「每個任務有自己的 QRCode，小老師掃碼直接進入被指派的任務」。這兩個架構的資料模型、權限設計、離線行為都不同。

**這個問題卡住**：幾乎所有 spec 的細節——進入流程、任務指派、離線同步、身份識別。

**決定結果**：

保留房間架構，一個班級一張 QRCode。小老師掃碼進入房間後，自己從任務清單中選擇要負責的任務（一個小老師可負責多個任務）。**不做任務專屬 QRCode**——讓老師為每個小老師合成對應的任務碼，會增加操作步驟，複雜度不值得。

讓小老師自行選擇任務而非系統或老師代為指定，符合「學生對自己的行為負責」的設計理念。

**資料模型影響**：任務（Task）需新增「指定的小老師」欄位（nullable）。非指定者仍可登記，但系統標記實際登記者的座號。相關 spec 見 spec.md User Story 2 與 Key Entities 更新。

取捨說明：小老師可能選錯任務（責任在學生）；所有小老師都能看見所有任務（班級事務本就公開，可接受）。

---

### 🔲 2. MVP 功能邊界

**問題**：目前哪些功能確定在 MVP、哪些不在，沒有明確的清單。

**這個問題卡住**：無法判斷一個 user story 是否在範圍內，也無法估計開發量。

**需要逐一確認的功能**：

| 功能                                | 在 MVP？ | 備註                |
| ----------------------------------- | -------- | ------------------- |
| 繳交狀況登記                        | ✅ 是    | 現有 spec 核心      |
| 成績登記                            | ✅ 是    | 新增 User Story 6   |
| 小老師身份識別（選座號 + 承諾裝置） | ✅ 是    | 已納入 User Story 2 |
| 小老師自行標記任務完成              | ✅ 是    | 新增 User Story 7   |
| 班級公開看板（學生可見）            | ❌ 否    | 移至下一階段        |
| 老師例外通知機制                    | ❌ 否    | 移至下一階段        |
| 家長端                              | ❌ 否    | 明確為後續版本      |

**決定結果**：MVP 包含繳交登記、成績登記、小老師身份識別、小老師標記任務完成。班級公開看板、老師通知機制、家長端均不在 MVP 範圍。

---

### ✅ 3. 「一般學生」角色定義

**問題**：系統目前定義了老師、小老師、家長三種角色，但非小老師的一般學生從未被說明。

**決定結果**：MVP 中一般學生不是系統使用者，不需進入系統。班級公開看板（一般學生唯讀介面）列為下一階段開發，屆時再定義此角色的進入方式與權限。

---

### ✅ 4. 老師通知的觸發條件

**問題**：vision.md 的設計目標是「老師是被通知後的決策者，不是主動監控者」，但觸發條件完全未定義。

**決定結果**：老師例外通知機制不在 MVP 範圍內。MVP 中老師仍需主動開啟系統查看狀況。通知機制的觸發條件設計移至下一階段，屆時再完整規劃。

---

### ✅ 5. 「任務」物件的欄位定義

**問題**：vision.md 多次提到「任務」，但它的欄位只能從敘述中零散拼湊。現有 data-model.md 是以「房間」為核心設計的，與 vision 的方向可能有出入。

**這個問題卡住**：資料模型更新、API 設計、離線同步邏輯。

**目前已知的任務欄位**：

- 登記項目類型（繳交 / 成績 / 其他）
- 截止時間（可彈性設定）
- 指派的小老師（可後來再改）
- 是否開放修改
- 任務狀態（進行中 / 小老師已標記完成 / 老師已結案）

**待確認的欄位**：

- 任務可以指派給多個小老師嗎？
- 任務的有效期間是一天還是可跨多天？
- 任務可以複製到下一天嗎（例如每日作業登記）？

**決定結果**：核心欄位已確立（見 spec.md Key Entities）。以下子問題不影響 MVP 開發，移至下一階段：任務是否可跨多天有效、任務是否可複製到下一天。

---

### 下一階段 spec 規劃

> 此區塊記錄本階段決定不在 MVP 內、但需要在下一次 spec 規劃時處理的項目。

- **老師例外通知機制**：系統在特定情況下主動通知老師（長時間未同步、任務截止前提醒、異常登記等），觸發條件待設計。目標是讓老師成為「被通知後的決策者」而非「主動監控者」。
- **班級公開看板**：讓一般學生（非小老師）和家長可以查看班級的登記狀況。需同時定義一般學生的進入方式、可見資料範圍（個人 vs. 全班）。
- **任務跨日與複製**：任務是否可設定多天有效期間；是否可一鍵複製為下一天的相同任務（例如每日作業登記）。
- **家長端功能**：家長接收通知、查看班級資訊的完整設計（參見 vision.md 第 6、7 節）。

<a name="全站網路同步狀態-policy"></a>

### 🔲 全站網路 / 同步狀態 policy

**問題**：網路連線狀態、同步狀態、警告觸發（裝置太久沒同步）等行為散落實作於：

- `src/components/NetworkStatus`、`src/components/SyncIndicator`（小老師端登記頁、任務清單頁底部）
- `src/hooks/useNetworkStatus`（小老師端散落使用）
- `src/lib/offline/syncController`（同步控制核心）

但沒有統一的 spec 描述以下事項：

1. **各畫面該顯示什麼狀態**：哪些 banner、何時顯示、何時消失、文案
2. **警告升級規則**：離線多久顯示「換個地方試試」、未同步多久通知老師（具體閾值）
3. **失敗 retry 策略**：指數退避？重試上限？放棄條件？
4. **老師端「裝置同步狀況」視圖**：002 班級狀況 tab（US4）的警告區會用到「裝置太久沒同步」訊號，但目前老師端完全沒有對應的視覺呈現與資料源
5. **/join 離線時的處理**：003 換座號流程把人推回 /join，必須統一處理「使用者在沒網路時抵達 /join」的情境
6. **i18n 對齊**：spec-align 目前不涵蓋 `src/i18n/messages.ts` 同步 → 使用者將擴充 spec-align skill 解決此缺口（不需 per-feature 額外文件）

**為什麼卡住**：影響 002 US4（班級狀況 tab 的警告區資料源）、003（小老師端入口 / 換座號 / 登記過程的網路提示），但本身不屬於任一 feature 範圍。

**處理方向**：開新 feature folder（例如 `specs/004-network-and-sync-policy/`）作為下一階段獨立議題。需先盤點現有元件實際行為，再對照 vision 第 6 節「老師是被通知後的決策者」決定警告升級規則。

**前置盤點**：需要做的事 ——

- 列出目前 `NetworkStatus` / `SyncIndicator` / `useNetworkStatus` / `syncController` 的實際行為
- 整理 001 spec.md 已有的相關 FR（FR-005 ~ FR-007、NFR-003）與 vision 期望
- 找出落差，整理成 004 的 spec.md

<a name="導師-vs-科任"></a>
### 🔲 導師 vs 科任老師的資料模型與流程支援

**問題**：台灣老師有兩種類型，目前 spec 與 data model 都隱含假設「一個老師管多個獨立班級」，但兩種類型實際需求很不同。

**導師（一個班級、所有科目）**：

- 主要動作：建立任務、管理學生、看異常
- Dashboard 只看一個班級
- 「我的班級」清單只有一筆
- **現有架構足以支援**，UIUX 上無需特殊調整

**科任（多班級、同一科目）**：

- 主要動作：**同一份任務通常要同時建立在多個班級**（例：五年甲乙丙的數學作業繳交）
- 跨班級報表最有價值（多班加總後問題才會集中浮現）
- Dashboard 看到的「班級」多個
- **現有架構運作但有顯著痛點**：必須一班一班手動建任務，重複勞動；無法跨班看「同一份任務」的總覽

**對 data model 的潛在影響**：

- 目前 `Task` 直接綁 `Room`（1:N）
- 科任老師「同一份任務套用到多個班級」→ 每班各生成一個 `Task` 物件 → 報表分散
- 可能解法：
  - **TaskTemplate** 抽象：老師建立範本 → 套用到多個班級 → 每班自動生成 Task，但記得 `templateId`
  - 跨班級報表以 `templateId` 聚合（例「五年甲乙丙的數學作業」總覽）
  - 報表 / 異常清單需要支援「按 templateId 分組」與「按 roomId 分組」兩種視角

**對任務 / 班級建立流程的影響**：

- 建立任務時可以多選班級（一次套用至多班）
- 指定座號可能要每班分別指定（座號 24 在不同班是不同人）
- 「快速建立多個班級」（例「我教五甲乙丙丁戊」一次建五個）值得考慮

**MVP 階段的處置**：

002 US8 已透過 UIUX 處理兩種老師：

- Dashboard 異常清單以「[班級名] · [任務名]」格式顯示，多班級老師可清楚識別歸屬
- 統計按跨班級加總呈現
- 側欄「我的班級」可展開清單作為多班級導航中樞

不在 MVP 範圍內：

- TaskTemplate 抽象 / 一次套用多班的任務建立流程
- 跨班級報表聚合（按 templateId）
- 一次建立多個班級

**處理方向**：開新 feature folder（例 `specs/005-teacher-types-and-templates/`）作為下一階段獨立議題，啟動前需先觀察 MVP 上線後科任老師實際的痛感程度。
