# Spec 前置問題清單 (Open Questions)

**文件用途**

這份文件列出在調整或新增 spec 之前，需要先做出決定的問題。每個問題都說明了它會卡住哪些 spec 工作。問題解決後打勾並記錄決定。

> 對照文件：`vision.md` 用來檢視方向對不對；這份文件用來確認決策做完了沒。

---

## 🔲 1. QRCode 機制：房間 vs. 任務（優先級：最高）

**問題**：現有 spec 的架構是「掃碼加入房間，房間內有多個登記項目」。vision.md 的成功場景描述的是「每個任務有自己的 QRCode，小老師掃碼直接進入被指派的任務」。這兩個架構的資料模型、權限設計、離線行為都不同。

**這個問題卡住**：幾乎所有 spec 的細節——進入流程、任務指派、離線同步、身份識別。

**需要決定的事**：
- 採用哪個架構？或是有第三種混合方式？
- 「房間」這個概念還需要嗎？還是被「任務」取代？
- 一個小老師可以同時被指派多個任務嗎？

## 🔲 2. MVP 功能邊界

**問題**：目前哪些功能確定在 MVP、哪些不在，沒有明確的清單。

**這個問題卡住**：無法判斷一個 user story 是否在範圍內，也無法估計開發量。

**需要逐一確認的功能**：

| 功能 | 在 MVP？ | 備註 |

| 繳交狀況登記 | ✅ 是 | 現有 spec 核心 |
| 成績登記 | ❓ | vision.md 說「會做」，但現有 spec 沒有 |

| 家長端 | ❌ 否 | 明確為後續版本 |

## 🔲 3. 「一般學生」角色定義

**問題**：系統目前定義了老師、小老師、家長三種角色，但非小老師的一般學生從未被說明。他們是系統使用者嗎？有介面嗎？班級公開看板是要讓他們查看的，但他們如何進入系統？

**這個問題卡住**：任何涉及「學生端」的 spec，以及班級公開看板的設計。

**需要決定的事**：
- 一般學生有沒有辦法進入系統？（掃碼？班級碼？）
- 一般學生進入系統後能做什麼？（唯讀？還是有操作？）
- 一般學生和小老師共用介面，還是分開？

## 🔲 4. 老師通知的觸發條件

**問題**：vision.md 的設計目標是「老師是被通知後的決策者，不是主動監控者」。這表示系統要在適當時機主動通知老師。但觸發條件完全未定義。

**這個問題卡住**：老師端的 spec 無法寫完整，因為老師「什麼時候要做事」的邏輯尚不清楚。

**需要決定的觸發條件（至少初步版本）**：
- 裝置超過多久沒同步要通知老師？
- 任務截止前多久要提醒老師查看？
- 有小老師登記異常（非指定者操作）時要通知嗎？
- 小老師標記任務完成時要通知老師嗎？

## 🔲 5. 「任務」物件的欄位定義

**問題**：vision.md 多次提到「任務」，但它的欄位只能從敘述中零散拼湊。現有 data-model.md 是以「房間」為核心設計的，與 vision 的方向可能有出入。

**這個問題卡住**：資料模型更新、API 設計、離線同步邏輯。

**目前已知的任務欄位**：
- 登記項目類型（繳交 / 成績 / 其他）
- 截止時間（可彈性設定）
- 指派的小老師（可後來再改）
- 是否開放修改
- 任務狀態（進行中 / 小老師已標記完成 / 老師已結案）

**待確認的欄位**：
- 任務可以指派給多個小老師嗎？
- 任務的有效期間是一天還是可跨多天？
- 任務可以複製到下一天嗎（例如每日作業登記）？
